01기획하기

먼저 무엇을 만들지 적습니다

좋은 앱은 코드가 아니라 기획에서 시작합니다. 개발자가 아니어도 괜찮아요. 시작 전에 무엇을 만들지 글로 정리해두는 것, 그게 첫 단추입니다.

PRD란?

Product Requirements Document, 즉 제품 요구사항 문서입니다. ‘무엇을, 왜, 어떻게 만들지’를 한곳에 적어둔 앱의 밑그림이에요. 화면, 기능, 사용자, 규칙을 미리 정리해 둡니다.

왜 쓰나?

AI에게 코드를 맡길 때 요구가 흐릿하면 AI도 헤매면서 토큰(AI가 읽고 쓰는 글자 단위)을 많이 쓰게 되고, 그만큼 크레딧(그 사용량에 매겨지는 요금)과 시간이 낭비됩니다. 기획이 뚜렷할수록 AI가 한 번에 정확히 만들어줘요. 제한된 크레딧과 촉박한 시간을 아끼는 첫 번째 단계입니다.

제가 실제로 쓰는 방법

저는 양식을 완벽히 채우기보다, 생성형 AI에게 만들고 싶은 앱을 장황하게 설명한 다음 “이걸 바탕으로 PRD로 정리해줘”라고 합니다. 대신 설명은 구체적이고 의도가 분명하게 — 고민되는 부분도 털어놓고, 넣고 싶은 기능도 하나하나 짚어가면서요.

이렇게 정리된 PRD는 저장해 둡니다. 나중에 다른 AI에서 다시 꺼내 쓸 수 있고, 복잡한 앱일수록 이 저장이 큰 힘이 되더라고요.

제가 느낀 장점은, AI가 정리하면서 제가 놓친 부분이나 부족한 곳을 짚어주고 더 나은 아이디어나 효율적인 방법을 제안해준다는 거예요. 좋으면 반영하면 됩니다.

02구조 이해하기

앱의 구조를 이해합니다

앱은 크게 두 부분입니다. 눈에 보이는 화면인 프론트엔드, 보이지 않는 곳에서 데이터를 저장·처리하는 백엔드. 백엔드가 얼마나 필요한지에 따라 앱의 종류가 갈립니다. 앞서 적은 4번 질문의 답을 여기 대입해 보세요.

서버 저장 없음 · 프론트만

03도구 결정하기

앱 유형에 맞는 도구를 고릅니다

무엇을, 어떤 구조로 만들지 정했다면 이제 도구입니다. 세 가지 앱 유형마다 필요한 도구가 다릅니다.

정적 앱

생성형 AI 하나면 충분

정적 앱은 생성형 AI만으로 충분합니다. 상위 요금제일수록 더 똑똑한(상위) 모델을 더 넉넉한 사용량으로 쓸 수 있어, 복잡한 요구도 더 안정적으로 처리해줘요.

시트 연동 앱

생성형 AI + 구글 스프레드시트

화면은 HTML로 만들고, 구글의 앱스 스크립트가 그 화면과 시트를 이어주는 백엔드 역할을 합니다. 데이터베이스 역할은 구글 시트가 하고요. AI 구독료 외 추가 비용이 없습니다.

무료 (서버·호스팅 비용 없음) 학교 계정으로 바로 접근 학운위 심의 부담 없음 데이터를 익숙한 시트로 확인·편집 동시 접속·실시간엔 약함 앱스 스크립트 실행 할당량 한계 데이터 많아지면 느려짐 학교 관리자 정책에 종속

※ ‘학운위 심의 부담 없음’은 학교가 이미 교육용 구글 도구를 심의·도입해 쓰고 있는 경우예요.

서비스형 앱

프론트 + 데이터베이스 + 코드 에디터(또는 IDE)

화면(프론트)과 파이어베이스 같은 데이터베이스가 필요합니다. 그리고 파일이 여러 개로 늘고 서로 얽히기 때문에, 그 파일들을 다룰 코드 에디터가 필요해요. 아래에서 하나씩 짚어볼게요.

꼭 파이어베이스여야 하나요?
아니요. 대표적 대안으로 Supabase가 있습니다. 파이어베이스가 자유로운 폴더 구조(NoSQL)라면, 수파베이스는 엑셀이나 구글 시트처럼 칸을 엄격히 나눠 표(SQL) 형태로 데이터를 관리하는 방식입니다. 교사 입장에선 한국어 자료와 예제가 훨씬 많은 파이어베이스가 무난한 출발점이에요.
코드 에디터란 무엇이고, 왜 필요한가요?
코드를 쓰고 고치는 전용 프로그램이에요. 워드가 문서 작성을 돕듯, 코드 에디터는 코딩을 돕습니다. 메모장에도 코드를 쓸 순 있지만, 코드 에디터는 색으로 구분해 구조를 한눈에 보여주고, 다음 코드를 자동완성해주고, 실행 전에 실수를 미리 짚어주고, 여러 파일을 한 화면에서 오가게 해줘요. 파일이 여럿 얽히는 서비스형 앱에선 사실상 필수입니다. 대표는 VS Code(요즘 표준)입니다.
통합 개발 환경(IDE)이란 무엇인가요?
IDE(Integrated Development Environment)는 코드 에디터에 ‘만드는 데 필요한 도구들’을 한데 묶은 거예요. 코드를 쓰는 에디터 + 실행해보는 기능 + 오류를 찾는 디버거 + 버전 저장(git) 같은 걸 한 프로그램에서 처리합니다. 코드 에디터가 ‘펜’이라면, IDE는 ‘펜·실행·검사·정리가 다 되는 작업대’예요. 요즘은 경계가 흐릿해서 VS Code도 확장 기능을 붙이면 IDE처럼 쓰이고, Cursor·Antigravity도 이 계열입니다.
요즘 코드 에디터는 뭐가 다른가요?
가장 큰 변화는 AI가 안으로 들어왔다는 거예요. 처음엔 다음 줄을 제안하는 ‘똑똑한 자동완성’ 수준이었는데, 지금은 AI 에이전트로 발전했습니다. 만들고 싶은 걸 말로 설명하면 AI가 여러 파일을 스스로 오가며 코드를 쓰고, 실행해보고, 오류가 나면 스스로 고치기까지 해요. Cursor·Antigravity·VS Code(Copilot) 모두 이 기능이 있습니다. 비개발자인 교사도 이 도구들을 쓸 수 있게 된 게 바로 이 덕분이에요 — 한 줄씩 짜는 대신 AI와 대화하며 만드니까요.

대표적으로 Cursor와 Antigravity가 있습니다. Cursor는 별도 유료 구독이 필요하고, Antigravity는 구글 AI(제미나이) 구독과 연동해 쓸 수 있으며 무료 등급도 있지만 사용량 제한이 빡빡한 편입니다. (요금 정책이 자주 바뀌니 사용 전 확인하세요.)

다만 생성형 AI와 함께 만들어 나갈 거라면 요금제 없이도 Cursor로 꽤 할 만합니다. 요즘 에디터에는 AI 에이전트 기능이 있어 바로 요구할 수도 있는데, 제 경험상 소통의 질이 아직 아쉽고 토큰·시간을 많이 쓰는 경우가 많았어요. 시행착오도 잦았고요. (이 부분은 빠르게 개선되는 중이라, 앞으로는 달라질 수 있습니다.) 그래서 저는 생성형 AI, 특히 Claude와 함께 작업하기를 추천합니다.

또 다른 길

도구를 조합하지 않고, 하나로 끝내기 — 올인원 AI 앱 빌더

지금까지는 ‘화면 만드는 도구 + 데이터베이스 + 배포’를 각각 챙기는 방식이었어요. 반면 Lovable 같은 올인원 AI 앱 빌더는 만들고 싶은 걸 말로 설명하면 화면·데이터베이스·로그인·배포까지 한 번에 만들어줍니다. 코드 에디터를 따로 열 필요 없이, 브라우저 안에서 전부 끝나요.

커서·안티그래비티와 같은 건가요?
아니요, 결이 다릅니다. 커서·안티그래비티는 코드를 편집하는 도구(에디터)라 데이터베이스와 배포는 내가 따로 챙겨야 해요. Lovable은 그 전 과정을 대신 해주고 서버(호스팅)까지 제공합니다. 실제로 Supabase 기반 백엔드와 호스팅을 함께 얹어줘요.
말로 설명하면 완성까지 빠름 데이터베이스·로그인·배포 원스톱 결과 코드를 GitHub로 내보내 소유 가능 크레딧 기반 유료 (무료 체험 있음) 복잡한 로직은 AI가 헤맬 수 있음 문제가 생기면 디버깅이 까다로울 수 있음

※ 무엇을 만드느냐(구조)와는 별개인 ‘만드는 방식’이에요. 복잡한 건 싫고 비용을 지불할 의향이 있다면 좋은 선택지입니다.

04배포하기

인터넷에 올립니다

배포는 ‘내 컴퓨터에만 있던 것’을 ‘누구나 열 수 있는 인터넷 주소’로 바꾸는 일입니다. 중요한 건, 앱 유형마다 올리는 곳이 다르다는 점이에요.

정적 앱

완성된 HTML 파일을 Netlify Drop · GitHub Pages · Vercel 중 한 곳에 올립니다. (아래 카드 참고)

시트 연동 앱

앱스 스크립트 편집기에서 ‘배포 → 웹 앱으로 배포’를 눌러 나오는 .../exec 주소가 곧 앱 주소입니다. Netlify에 끌어다 놓는 방식이 아니에요.

서비스형 앱

보통 Firebase Hosting 또는 Vercel에 올립니다. 데이터베이스(파이어베이스 등)와 함께 동작하도록 연결합니다.

가장 쉬움

Netlify Drop

정적 앱에 딱 맞아요

app.netlify.com/drop 접속 (로그인은 나중에 해도 됩니다)

index.html이 든 폴더를 통째로 끌어다 놓기

~.netlify.app 주소가 생성 — 바로 공유하면 끝!

팁 — 폴더 안에 index.html이 꼭 있어야 하고 50MB 이하면 안정적입니다.

완전 무료

GitHub Pages

무료·안정적, 학교에 부담 없음

GitHub에 저장소(repository)를 만들고 파일 올리기

저장소 Settings → Pages에서 브랜치 선택

~.github.io 주소가 생성됩니다

팁 — 첫 설정에 GitHub 계정과 약간의 적응이 필요하지만, 한 번 익히면 완전 무료로 오래 쓸 수 있어요.

커지면 이걸로

Vercel

프로젝트가 커지고 자주 고칠 때

vercel.com/drop에 파일·폴더·zip 끌어다 놓기

팀과 프로젝트 이름을 고르고 Deploy

GitHub와 연결하면 수정할 때마다 자동 배포

팁 — 프레임워크 프로젝트나 바이브코딩 결과물도 알아서 빌드해줍니다.

⚠ 배포하기 전에 — 보안, 이건 꼭 확인하세요
  • 01배포하면 화면 코드는 누구나 열어볼 수 있습니다. ‘내 컴퓨터에서만 보이던 것’이 아니에요. 이 전제를 항상 염두에 두세요.
  • 02유료 API 키(예: Gemini API 키)는 절대 화면 코드에 넣지 마세요. 노출되면 요금 폭탄으로 이어집니다. 이런 키가 필요한 처리는 앱스 스크립트나 백엔드 뒤에 숨겨야 합니다.
  • 03파이어베이스 키는 그 자체로 비밀은 아니지만, 보안 규칙(Security Rules)이 반드시 필요합니다. 규칙 없이 두면 누구나 데이터를 읽고 지울 수 있어요.
  • 04학생 데이터는 최소한만 수집하고 익명화하세요. 실명 대신 학급·번호 기반의 익명 코드(예: 309lion)를 쓰면 사고가 나도 피해가 작습니다.

06디버깅

고치는 시간이 만드는 시간보다 깁니다

디버깅은 만든 것이 의도대로 작동하지 않을 때 원인을 찾아 고치는 과정입니다. 여기서 bug는 벌레·결함을 뜻하고(옛날 컴퓨터에 낀 나방 일화가 유명하죠), 디버깅은 그 결함을 잡아내는 일이에요. 앱을 만들 때 가장 오래 걸리는 부분이 바로 여기입니다. 순서만 알아두면 훨씬 수월해져요.

증상을 정확히 봅니다. 무엇이 어떻게 잘못됐는지 관찰하세요. 화면이 안 뜨나요? 버튼이 안 눌리나요? 숫자가 틀리나요?

단서를 모읍니다. 브라우저에서 F12(개발자 도구)를 열고 ‘콘솔’에 뜨는 빨간 오류 메시지를 확인하세요. 가장 중요한 단서입니다.

AI에게 넘깁니다. 그 오류 메시지와 관련 코드를 평소 쓰던 생성형 AI(Claude, ChatGPT, Gemini 등)에 그대로 붙여넣고 “왜 이 오류가 나는지, 어떻게 고치는지” 물어보세요. 툴마다 해결책이 다를 수 있으니 여러 곳에 교차 검증해보는 것도 좋습니다.

한 번에 하나씩 고칩니다. 여러 곳을 동시에 바꾸지 마세요. 하나 고치고 → 확인 → 또 하나. 그래야 무엇이 문제였는지 알 수 있습니다.

범위를 좁힙니다. 문제가 어디서 나는지 절반씩 잘라가며 확인해 원인 지점을 좁혀갑니다.

팁 — 오류는 없는데 이상하게 동작한다면, 중간중간 값을 찍어보며(console.log) 어디서부터 어긋나는지 추적하세요.

만들었다면, 책임도 함께

학생은 미성년자이고, 우리가 만드는 앱은 그 데이터를 다룹니다. 01 기획에서 ‘개인정보를 받나?’에 ‘예’라고 답했다면, 이 단계는 선택이 아니라 필수입니다. 곁다리가 아니라, 앱을 세상에 내놓기 위한 조건이에요. 이 부분은 다음 시간에 함께 자세히 다룹니다.
